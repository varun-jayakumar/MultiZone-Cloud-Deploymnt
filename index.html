  • Webapp: Contains the web application (REST API) code. GitHub Actions are used for Continuous Deployment (CD):
    • When code is merged, an AMI of the webapp is built on a VM spun up on Google Cloud, using Packer.
    • The built AMI is then used to create an instance template on Google Cloud.
    • The new instance template refreshes the existing instance group.
  • Terraform-IAC: Contains Infrastructure as Code (IAC) for setting up and modifying infrastructure.
  • Serverless: Contains JavaScript function code deployed on Google Cloud Function to send verification emails when a user signs up.

Infrastructure Components

  • Domain name purchased from Namecheap.
  • Public DNS zone hosted on Google Cloud with an A record pointing to the Load Balancer IP.
  • SSL certificate attached to the IP (domain), generated by Google Cloud.
  • The Load Balancer routes traffic to the instance group deployed from the custom AMI instance template.
  • Autoscaling and health checks are configured for the instance group, which is deployed across three zones in the US-East1 region.
  • The VPC contains private and public subnets, with all resources created in the US-East1 region.
  • VMs are deployed in a public subnet, allowing only SSH (port 22) and traffic from the Load Balancer (port 3000).
  • SQL DB instance deployed in Google's VPC, accessible from the VMs through VPC peering.

Event-Driven Components

  • Pub/Sub: When the /v1/user (create user) endpoint is hit, a message is posted to a Pub/Sub queue. This message triggers a Google Cloud Function:
    • The Cloud Function sends a verification email to the user with a link that expires in 2 minutes.
  • Private Google Access is used within the VPC for VMs to push messages into the Pub/Sub queue.
  • Bucket (gcf-email-service) stores artifacts of the serverless repository for deployment.
  • Customer-managed key created and stored in a key ring used to encrypt VM disks, SQL instance, and S3 bucket containing serverless artifacts.
